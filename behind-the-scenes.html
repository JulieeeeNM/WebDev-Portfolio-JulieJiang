Behind the Scenes

This page provides a look at how I built this portfolio website, including technical challenges, design decisions, and lessons learned throughout the development process.

Development Process

Building this portfolio involved several key phases:

Planning & Wireframing

I started by sketching wireframes and creating a sitemap to plan the structure and content hierarchy.

HTML Structure

Using semantic HTML5 elements was a priority to ensure accessibility and proper document structure.

<header>
    <nav>
        <ul>
            <li><a href="index.html">Home</a></li>
            <!-- Other navigation items -->
        </ul>
    </nav>
</header>
<main>
    <section aria-labelledby="section1-heading">
        <h2 id="section1-heading">My Projects</h2>
        <!-- Content -->
    </section>
</main>

CSS Implementation

I used a mobile-first approach with CSS Grid and Flexbox for responsive layouts. CSS variables helped maintain consistency.

:root {
    --primary-color: #2c3e50;
    --secondary-color: #3498db;
    --accent-color: #e74c3c;
    --text-color: #333;
    --light-bg: #f9f9f9;
}

.portfolio-grid {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(300px, 1fr));
    gap: 2rem;
}

@media (max-width: 768px) {
    .portfolio-grid {
        grid-template-columns: 1fr;
    }
}

JavaScript Features

Interactive elements like the modal windows and form validation were implemented with JavaScript.

// Modal functionality
function openModal(modalId) {
    const modal = document.getElementById(modalId);
    modal.style.display = "block";
    document.body.style.overflow = "hidden";
}

// Form validation
function validateForm() {
    const email = document.getElementById('email').value;
    if (!email.includes('@')) {
        showError('emailError', 'Please enter a valid email address');
        return false;
    }
    return true;
}

Challenges & Solutions

Responsive Navigation

Challenge: Creating a navigation that works well on both desktop and mobile devices while remaining accessible.

Solution: Implemented a mobile menu toggle with proper ARIA attributes and keyboard navigation support.

<button class="menu-toggle" aria-label="Toggle navigation menu" 
        aria-expanded="false" aria-controls="main-nav">
    ☰
</button>
<nav id="main-nav">
    <!-- Navigation items -->
</nav>

// JavaScript to toggle menu
menuToggle.addEventListener('click', function() {
    const expanded = this.getAttribute('aria-expanded') === 'true';
    this.setAttribute('aria-expanded', !expanded);
    navMenu.classList.toggle('visible');
});

Project Filtering

Challenge: Allowing users to filter portfolio items by category without page reloads.

Solution: Used JavaScript to filter items based on data attributes, with smooth transitions.

// Filter portfolio items
filterButtons.forEach(button => {
    button.addEventListener('click', () => {
        const filter = button.dataset.filter;
        
        portfolioItems.forEach(item => {
            if (filter === 'all' || item.dataset.category === filter) {
                item.style.display = 'block';
            } else {
                item.style.display = 'none';
            }
        });
    });
});

Form Validation

Challenge: Providing clear validation feedback while maintaining accessibility.

Solution: Implemented client-side validation with descriptive error messages tied to form fields.

function showError(elementId, message) {
    const errorElement = document.getElementById(elementId);
    errorElement.textContent = message;
    errorElement.style.visibility = 'visible';
    document.getElementById(elementId.replace('Error', ''))
             .setAttribute('aria-invalid', 'true');
}

// Clear errors on input
inputs.forEach(input => {
    input.addEventListener('input', () => {
        const errorId = input.id + 'Error';
        document.getElementById(errorId).style.visibility = 'hidden';
        input.setAttribute('aria-invalid', 'false');
    });
});

Key Lessons Learned

Mobile-First Approach

Starting with mobile design forced me to prioritize content and resulted in a cleaner, more focused layout.

Progressive Enhancement

Building core functionality first and then enhancing it ensured the site remained usable without JavaScript.

Accessibility from Start

Incorporating accessibility considerations from the beginning was easier than retrofitting later.

CSS Organization

Using CSS variables and modular stylesheets made maintenance much easier as the project grew.

Future Improvements

Areas I plan to enhance in future iterations:

  • Implement a dark mode toggle
  • Add more interactive elements with smooth animations
  • Integrate a blog section with a CMS
  • Optimize performance further with lazy loading
  • Expand accessibility testing with more assistive technologies
